Editor → Production — Abcotronics ERP

Pipeline overview for your stack: Cursor (or any editor) → Git → GitHub → DigitalOcean droplet (Nginx, PM2, Node server.js) → Prisma → managed PostgreSQL. Production domain referenced in repo docs: abcoafrica.co.za.

Overview

Nothing reaches users until it is committed, pushed to GitHub, deployed to the server, and the Node process is restarted. The application is a single Node/Express process that serves both /api/* and the SPA/static assets. Persistent data lives in PostgreSQL; if the database is unreachable, API routes may return 503 or time out even when HTML still loads.

1. Editor (Cursor or any IDE)

What it is

Where source files exist on your computer until you commit.

What it is for

You change behaviour and UI here: React/JSX under src/, HTTP handlers under api/, Express entry server.js, schema and migrations under prisma/. Production does not change until later steps.

How to implement

  • Use a feature branch for non-trivial work (feature/…, fix/…).
  • Run the live app locally as this repo expects: npm run dev:backend and npm run watch:jsx.
  • The separate frontend/ tree is a separate modernization project; the production ERP UI is built from src/ per project conventions.
  • Exercise the flows you care about (e.g. job cards, /api/me) before pushing.

2. Git commit

What it is

A snapshot of the repository at a point in time, with a message and a commit hash.

What it is for

History, rollback, collaboration, and a fixed point that deployment can target.

How to implement

git status
git add -p   # or add specific paths
git commit -m "Clear description of the change"

3. Git push

What it is

Uploading local commits to the remote on GitHub.

What it is for

Off-machine backup, sharing with others, and the source that your droplet’s git pull uses.

How to implement

git push origin your-branch
# After merge to main:
git checkout main && git pull && git push origin main

If you push only to a branch that is never merged or deployed, production stays on old code.

4. GitHub (remote repository)

What it is

The authoritative hosted copy: branches, pull requests, tags.

What it is for

Single source of truth, optional PR/review workflows. This repository does not include .github/workflows by default, so CI/CD must be defined elsewhere if used.

How to implement

Merge to the branch you actually deploy (often main). Optionally tag releases for traceability.

5. Deploy (server update)

What it is

Fetching latest code on the droplet, installing dependencies, building if required, running migrations when the schema changes, restarting PM2.

What it is for

Users hit the server, not your laptop. Restart loads new code and refreshed env when using --update-env.

How to implement (typical pattern)

ssh root@abcoafrica.co.za   # or root@YOUR_DROPLET_IP
cd /var/www/abcotronics-erp   # adjust to your server path
git fetch origin && git checkout main && git pull
npm ci                        # or npm install — be consistent with prod
npm run build                 # when you ship built JSX/CSS to dist/
npx prisma migrate deploy     # when prisma schema changed — per your release process
pm2 restart abcotronics-erp --update-env

You may use existing scripts in the repo (e.g. deploy helpers) instead of typing these by hand—keep one documented path and stick to it.

6. Droplet: Nginx → PM2 → Node

Component Role
Nginx Listens on 443 (HTTPS), terminates TLS, proxies to Node (often http://127.0.0.1:3000).
PM2 Runs node server.js under the process name (e.g. abcotronics-erp), restarts on failure, manages logs.
Node / Express One process: REST under /api/*, Prisma, plus SPA and static files (index.html, dist/).

Verify

  • On the droplet: curl -sS http://127.0.0.1:3000/ or your documented health/db endpoint.
  • pm2 status, pm2 logs abcotronics-erp.
  • nginx -t and confirm proxy_pass matches the Node port.

7. PostgreSQL (DigitalOcean managed)

What it is

Managed Postgres; Prisma uses DATABASE_URL from production environment.

What it is for

All durable ERP data. Connection failures, timeouts, or pool exhaustion surface as API errors (e.g. 503 on /api/me).

How to implement

  • Set DATABASE_URL in production .env beside server.js (never commit secrets).
  • Restrict DB network access to trusted hosts (e.g. droplet only).
  • Apply migrations when schema changes; monitor connection limits in DO and in logs.

End-to-end chain (one sentence)

Edit locally → commit → push → merge on GitHub → deploy on the droplet (pull, install, build, migrate if needed, PM2 restart) → Nginx serves HTTPS to Express → Prisma reads/writes PostgreSQL.

Post-deploy checklist

  • Site loads in browser; hard refresh after deploy.
  • Authenticated API checks succeed when DB is healthy (e.g. /api/me).
  • PM2 logs show no crash loops; no repeated DB connection errors.
